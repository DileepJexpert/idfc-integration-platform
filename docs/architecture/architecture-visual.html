One engine, many doors — and why a conductor runs the show

How a loan request travels from a partner or Salesforce, through a data-driven orchestration engine, out to the bank's real systems, and safely back — plus the two patterns that keep it correct when money is involved: the saga (safe undo) and orchestration (a central conductor instead of scattered signalling).

01High level · the whole platform

Every channel enters through a thin door, then runs the same core

A new channel is a new door, never a fork of the core. Each door normalises its input into one shared envelope and hands it to the engine; the engine calls reusable capabilities; capabilities call the bank's real vendors behind swappable ports.

The load-bearing idea

N channels × M products would be N×M things to build. By normalising every door into one envelope and one engine, it becomes N + M: add a channel = a door; add a product = a journey. Nothing else changes.

02Two lanes, one capability model

Long journeys go async; money movement stays synchronous

Async engine lane

Multi-step business journeys

Loan origination, device validation, RC verification. The caller gets a fast acknowledgement; the work runs later over Kafka as a durable, auditable journey.

  • Walks a DAG of nodes
  • Durable run state in Aerospike — resumable on crash
  • Every step visible in the ops view
✓ answer comes back later · fully audited
Sync lane

Real-time request / response

IMPS fund transfer (loan disbursal), LMS offer check. The caller blocks and gets the answer in the HTTP response — you cannot fire-and-forget a money transfer.

  • Runs in-thread — no engine, no Kafka
  • Idempotent: a retry never double-pays
  • A timeout is AMBIGUOUS, never a false success
✓ answer is immediate · no journey run

05Pattern · the saga

Saga — how you undo a half-done loan safely

You cannot hold one database transaction across the bank's core systems (FinnOne, bureaus, KYC). A saga replaces that impossible distributed lock with a chain of local steps, each with a matching compensation that undoes it. If a later step fails, completed steps are reversed — in reverse order.

The use case we solve

Booking a loan has a real side-effect

When n_book runs, a loan record is created in FinnOne — real money committed. If anything downstream can't complete, the platform must not just error out and leave a phantom loan on the books. It must reverse the booking.

The journey declares this inline on the node:

  • onFailure: "compensate" — start the saga
  • compensation: reverseBooking(loanId) — the undo
  • meter: finnone_pool (max 20) — throttle concurrent bookings so the core isn't flooded

  • The business outcome is decided immediately (the caller is told now)

Why it matters

No phantom loans, no double disbursals, no 2am "is this loan real?" — the undo is declared next to the step it undoes, and the engine guarantees it runs exactly once.

06Pattern · orchestration vs choreography

Two ways to coordinate services — we chose the conductor

When many services must cooperate on one outcome, there are two styles. The difference is whether a central brain holds the flow, or the flow is an emergent side-effect of who reacts to what.

Choreography

Dancers, no conductor

⇢ event ⇢

⇢ … ⇢

Each service listens for an event, does its bit, emits the next event. No one owns the whole flow.

  • Loose coupling, no central component
  • ✗ the flow is implicit — nowhere to read "what happens to a loan"
  • ✗ each service must know what comes next (hidden coupling)
  • ✗ compensation logic scattered across services
  • ✗ "where is loan X?" has no single answer

Why orchestration is the right call for regulated lending
The needWhy orchestration wins here
AuditabilityA regulator asks "what happened to this application, which step, why declined?" — the engine records every hop; choreography keeps no central record.
Change velocityProducts & policies change constantly — the flow is data (a DAG in the Designer), so business changes it without a code release.
Safe undoReversing a booking across systems needs a coordinator — the engine runs the saga in reverse order, exactly once.
Safety by defaultAn unknown request type is quarantined to a dead-letter queue (fail-closed), never silently mis-processed.
ReuseThe same KYC / bureau / scoring capabilities serve every product because they carry no flow knowledge.

Choreography suits high-throughput, loosely-related events (notifications, analytics fan-out). A regulated, multi-step, money-committing loan flow wants the conductor.
